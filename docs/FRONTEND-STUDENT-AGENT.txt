БРИФ ДЛЯ ФРОНТЕНД-АГЕНТА: ПАНЕЛЬ СТУДЕНТА (Next.js) + БЭКЕНД robomind-back
================================================================================

ЦЕЛЬ ДОКУМЕНТА
--------------
Ты разрабатываешь клиентскую часть для УЧЕНИКА (панель студента). Данные и
бизнес-логика живут в отдельном репозитории бэкенда — NestJS (robomind-back).
Этот файл объясняет структуру бэкенд-проекта и контекст продукта. Детальный
контракт HTTP для панели студента — в ОТДЕЛЬНОМ файле (что передавать,
что приходит в ответ, ошибки):

  docs/API-STUDENT-PANEL.md

ЧТО ЗА ПРОДУКТ
--------------
Платформа курсов по робототехнике для школьников: регистрация (ИИН, ФИО, школа),
доступ к курсам выдаёт админ, модули с видео/текстом/файлами/эфиром, после
теста открывается следующий модуль, дашборд прогресса, ИИ-ассистент по модулю,
сертификаты. Панель студента — только сценарии УЧЕНИКА (не админка).

СТЕК БЭКЕНДА (robomind-back)
-----------------------------
- Node.js, NestJS 11
- PostgreSQL + TypeORM, миграции в src/database/migrations/
- Модули API разбиты по зонам:
  * auth          — вход, регистрация (общий для студента и админов)
  * app-api       — всё, что нужно фронту студента (префикс /app/...)
  * admin-api     — админка (панель студента НЕ вызывает, кроме редких исключений)
  * ai            — ИИ для студента: /app/ai/...
  * device        — лимит устройств, уведомления админам (логика на бэке при логине)

КОРНЕВОЙ URL API
----------------
Все запросы:  {BACKEND_ORIGIN}/api/v1/...
Пример:       https://api.example.com/api/v1/auth/login

СТРУКТУРА РЕПОЗИТОРИЯ БЭКЕНДА (кратко)
---------------------------------------
robomind-back/

  docs/
    API-STUDENT-PANEL.md    — АПИ ТОЛЬКО ДЛЯ ПАНЕЛИ СТУДЕНТА (используй в первую очередь)
    REST-API.md             — полный список маршрутов (включая админку)
    DEVICE-LIMIT.md         — лимит устройств при входе
    AI-SERVICES.md          — ИИ-сервисы OpenAI

ВАЖНО ДЛЯ СТУДЕНТА: DEVICE ID
----------------------------
При логине ученик ОБЯЗАН передавать стабильный UUID устройства (один на
браузер/приложение, localStorage или secure storage). Максимум 2 устройства на
аккаунт. Третье — 403, админы школы и супер-админ получают уведомление.
Подробности: docs/DEVICE-LIMIT.md

АВТОРИЗАЦИЯ (текущее и целевое состояние)
-----------------------------------------
Сейчас часть эндпоинтов для разработки использует заголовок x-user-id (UUID).
Целевой вариант: JWT — Authorization: Bearer <access_token> после реализации
на бэке. Панель студента должна хранить токены безопасно и подставлять Bearer.

РОЛЬ ФРОНТА ПАНЕЛИ СТУДЕНТА
--------------------------
1) Онбординг / регистрация: каскад город → район → школа, форма данных.
2) Логин с deviceId.
3) Список доступных курсов, модули, плеер контента, тесты.
4) Дашборд: прогресс, сертификаты, ИИ-рекомендации (когда готово).
5) ИИ-чат по текущему модулю (лимит запросов в день на бэке).
